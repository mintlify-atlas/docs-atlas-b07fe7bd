## Overview

Metis is a pre-planning consultant named after the Greek goddess of wisdom, prudence, and deep counsel. She analyzes user requests BEFORE planning to prevent AI failures by identifying hidden intentions, detecting ambiguities, and flagging potential AI-slop patterns.

**Identity**: Strategic advisor who asks clarifying questions and prepares directives to guide the planner agent toward successful implementation.

## When to Use Metis

### Recommended Scenarios

<Check>**Ambiguous requests** - User request lacks clear requirements or scope</Check>
<Check>**Complex tasks** - Non-trivial work requiring upfront planning</Check>
<Check>**Open-ended features** - "Add feature" without specifics</Check>
<Check>**Refactoring work** - Ensure behavior preservation is defined</Check>
<Check>**Architecture decisions** - Strategic choices requiring analysis</Check>

### Avoid Metis For

<Warning>**Simple, well-defined tasks** - "Fix typo in README", "Add console.log"</Warning>
<Warning>**Already detailed requirements** - User provided comprehensive specifications</Warning>
<Warning>**Trivial changes** - Minor tweaks don't need pre-planning</Warning>
<Warning>**Urgent fixes** - When speed matters more than perfect planning</Warning>

## Critical Rules

Metis follows strict analysis principles:

### NEVER

- Skip intent classification
- Ask generic questions ("What's the scope?")
- Proceed without addressing ambiguity
- Make assumptions about user's codebase
- Suggest acceptance criteria requiring user intervention
- Leave QA criteria vague or placeholder-heavy

### ALWAYS

- Classify intent FIRST
- Be specific ("Should this change UserService only, or also AuthService?")
- Explore before asking (for Build/Research intents)
- Provide actionable directives for Prometheus
- Include QA automation directives in every output
- Ensure acceptance criteria are agent-executable

## Usage Examples

### Example 1: Build from Scratch

```typescript
// User request
"Add user authentication to the API"

task(
  subagent_type="metis",
  description="Analyze authentication requirements",
  prompt="User requested: Add user authentication to the API. 
          Perform pre-planning analysis."
)

// Metis response:
"## Intent Classification
**Type**: Build from Scratch
**Confidence**: High
**Rationale**: New feature, no existing auth system mentioned

## Pre-Analysis Findings

Launched explore agent to find existing patterns:
- Found Express.js app in `src/server.ts`
- No existing auth middleware
- Current endpoints: `/api/users`, `/api/posts` (no protection)
- Uses PostgreSQL for data storage

## Questions for User

1. **Authentication method**: JWT tokens, sessions, or OAuth?
   - JWT is stateless and works well with APIs
   - Sessions require session storage (Redis)
   - OAuth delegates to third-party (Google, GitHub)

2. **What endpoints need protection?**
   - All `/api/*` routes?
   - Only specific routes (POST/PUT/DELETE)?
   - Public read, protected write?

3. **User registration flow**:
   - Email/password signup?
   - Email verification required?
   - Social login options?

4. **Password requirements**:
   - Minimum length, special characters?
   - Or follow existing company policy?

5. **Scope boundaries** (AI-slop prevention):
   - Add authentication ONLY, or also:
     * Authorization (roles/permissions)?
     * Password reset flow?
     * Rate limiting?
     * Refresh tokens?

## Identified Risks

- **Security misconfiguration**: JWT secrets, bcrypt rounds, token expiry
  - Mitigation: Use environment variables, follow OWASP guidelines

- **Backward compatibility**: Existing API clients may break
  - Mitigation: Define migration strategy (grace period with opt-in?)

- **Scope creep**: Auth often expands to roles, permissions, MFA
  - Mitigation: Define explicit exclusions upfront

## Directives for Prometheus

### Core Directives
- MUST: Follow Express middleware pattern from `src/server.ts:15-30`
- MUST: Store passwords with bcrypt (12 rounds minimum)
- MUST: Use environment variables for secrets (never hardcode)
- MUST NOT: Add authorization (roles/permissions) unless explicitly requested
- MUST NOT: Implement password reset flow unless explicitly requested
- MUST NOT: Add OAuth providers unless explicitly requested
- PATTERN: Follow error handling pattern from `src/middleware/error-handler.ts`

### QA/Acceptance Criteria Directives (MANDATORY)
- MUST: Write executable tests, not manual steps
- MUST: Use curl commands for API verification
- MUST: Include exact expected responses (JSON structure, status codes)
- MUST NOT: Require "user manually tests login page"
- MUST NOT: Use placeholders like "[your-token]" without concrete example

Example of GOOD acceptance criteria:
```bash
# Register new user
curl -X POST http://localhost:3000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"TestPass123!"}'
# Expected: {"success":true,"userId":1}

# Login
curl -X POST http://localhost:3000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"TestPass123!"}'
# Expected: {"token":"eyJhbGc...","expiresIn":3600}

# Access protected endpoint
curl http://localhost:3000/api/users \
  -H "Authorization: Bearer eyJhbGc..."
# Expected: [{"id":1,"email":"test@example.com"}]
```

## Recommended Approach

1. Get user answers to questions 1-5
2. Consult librarian for JWT best practices
3. Prometheus creates implementation plan following directives
4. Plan must include executable acceptance tests"
```

### Example 2: Refactoring

```typescript
// User request
"Refactor the authentication service to use a cleaner structure"

task(
  subagent_type="metis",
  description="Analyze refactoring requirements",
  prompt="User requested: Refactor the authentication service. 
          Perform pre-planning analysis."
)

// Metis response:
"## Intent Classification
**Type**: Refactoring
**Confidence**: High
**Rationale**: Explicit refactor request, existing code modification

## Pre-Analysis Findings

Analyzed existing authentication service:
- `src/services/auth.ts` (450 lines, multiple responsibilities)
- Handles: token generation, password hashing, user validation, session management
- Tests exist: `src/services/auth.test.ts` (28 tests, all passing)

## Questions for User

1. **Behavior preservation** (CRITICAL for refactoring):
   - Must all 28 tests continue passing without modification?
   - Or can we update tests if behavior intentionally changes?

2. **Refactoring scope**:
   - Restructure `auth.ts` only?
   - Or also related files (middleware, routes)?

3. **What specific structure issues?**:
   - Too many responsibilities in one file?
   - Unclear function naming?
   - Hard to test?
   - Performance problems?

4. **Rollback strategy**:
   - Can we use feature flag to toggle old/new implementation?
   - Or is direct replacement acceptable?

## Identified Risks

- **Regression introduction**: Changing working code may break subtle behaviors
  - Mitigation: Run full test suite after EACH file change, not just at end

- **Adjacent code changes**: Refactor may require updating callers
  - Mitigation: Use `lsp_find_references` to map all usages first

- **Scope creep**: "Clean structure" may expand to adding new features
  - Mitigation: Define "Must NOT change behavior" directive

## Directives for Prometheus

### Core Directives
- MUST: Define pre-refactor verification (exact test command + expected output)
- MUST: Use `lsp_find_references` to map all usages of functions being moved
- MUST: Verify tests pass after EACH change, not just at the end
- MUST: Use `lsp_rename` for safe symbol renames across codebase
- MUST NOT: Change behavior while restructuring
- MUST NOT: Add new features during refactoring
- MUST NOT: Refactor adjacent code not explicitly in scope
- TOOL: Use `ast_grep_replace(dryRun=true)` to preview transformations

### Pre-Refactor Verification
```bash
bun test src/services/auth.test.ts
# Expected: All 28 tests pass
```

### QA/Acceptance Criteria Directives
- MUST: All original tests continue passing
- MUST: Run tests after each intermediate commit
- MUST: Verify authentication flow still works:
```bash
curl -X POST http://localhost:3000/api/login \
  -H "Content-Type: application/json" \
  -d '{"email":"existing@user.com","password":"ExistingPass123"}'
# Expected: Same response format as before refactoring
```

## Recommended Approach

Refactoring should be incremental with verification at each step:
1. Run and document pre-refactor baseline
2. Use LSP tools to map dependencies
3. Create plan with verification after each change
4. NO behavior changes - structure only"
```
